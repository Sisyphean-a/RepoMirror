# RepoMirror

`RepoMirror` 是一个基于 `Go + Wails + React` 的 Windows 桌面工具，用于在两个本地 Git 仓库之间执行单方向镜像同步。

它不是 Git 客户端，也不是代码合并工具，而是一个把“源仓库当前可同步内容”镜像到“目标仓库”的专用工作台。用户可以在 `A → B` 和 `B → A` 两个方向之间切换，但任一时刻只处理一个明确方向。

---

## 1. 项目定位

这个项目解决的是一种很具体的工作场景：

- 同一套代码需要在两个仓库之间持续搬运
- 用户希望看到“真正会被同步的文件差异”，而不是抽象的 Git 概念
- 用户需要在同步前确认目标仓库当前是否干净、是否已有未提交改动
- 用户希望同步、提交、推送在同一界面内完成，不在多个工具之间跳转

因此，`RepoMirror` 的定位不是“功能全面”，而是“路径清晰、风险显式、边界明确”。

---

## 2. 当前具体功能

### 2.1 仓库选择与配置持久化

- 支持选择仓库 `A` 与仓库 `B`
- 选中的目录会先解析为真实 Git 仓库根目录，而不是盲目保存用户点击的路径
- 支持保存当前配置
- 应用会持久化以下信息：
  - 仓库 A 路径
  - 仓库 B 路径
  - 当前同步方向
  - 窗口宽高

当前配置文件默认保存在：

- `%APPDATA%/RepoMirror/config.json`

### 2.2 双方向切换，但单次只处理一个方向

- 支持 `A → B`
- 支持 `B → A`
- 支持“一键交换仓库 A / B 路径”

这里的设计重点是：

- “双向能力”只体现在方向可切换
- “执行动作”始终是单向镜像，不做双向自动合并

这能避免用户把它误解成同步盘或 Git merge 工具。

### 2.3 差异计算

系统会根据当前方向，识别：

- 谁是源仓库
- 谁是目标仓库
- 哪些文件应该进入同步候选集
- 哪些文件虽然不同，但由于目标端保护规则存在，不能被覆盖或删除

差异结果分为两层：

- `新增`
- `修改`
- `删除`
- `受保护`（会显示，但不会参与同步）

并在界面中给出：

- 差异总数
- 各类型计数
- 可筛选的差异列表

### 2.4 同步执行

点击“执行同步”后，系统会把当前源端可同步文件镜像到目标仓库：

- 源端存在、目标端不存在的文件会被复制
- 源端与目标端内容不同的文件会被覆盖
- 源端不存在、目标端存在且允许处理的文件会被删除

同步完成后会立即刷新界面状态，而不是停留在旧快照上。

### 2.5 目标仓库状态检查

界面右侧持续展示当前目标仓库状态，包括：

- 仓库名
- 当前分支
- 是否干净
- 未提交文件数量
- 未跟踪文件数量

如果目标仓库不是合法 Git 仓库，或者 Git 状态读取失败，界面会直接显示错误信息，不做静默降级。

### 2.6 提交与推送

对于当前目标仓库，支持直接执行：

- `git add -A`
- `git commit -m "<message>"`
- `git push`

界面中提供提交信息输入框与两个动作按钮：

- `提交`
- `推送`

这里的约束也很明确：

- 提交和推送只作用于当前目标仓库
- 不会影响源仓库
- 提交信息为空时会直接报错，不会自动生成默认文案

## 3. 同步规则与硬边界

这是当前产品最关键的行为定义，也是 UI/UX 评估时需要重点关注的部分。

### 3.1 明确保护的内容

以下内容永远不会参与同步：

- `.git` 目录
- 任意层级的 `.gitignore` 文件
- 被目标仓库 `git check-ignore` 命中的路径

这意味着：

- 工具不会破坏 Git 元数据
- 不会把源仓库里的 `.gitignore` 覆盖到目标仓库
- 不会强行写入目标仓库明确声明应忽略的内容

### 3.2 不做复杂 Git 工作流

当前产品明确不实现以下能力：

- merge
- rebase
- stash
- conflict resolve
- branch 管理
- commit 历史浏览

这不是缺失，而是主动裁剪。项目只承担“镜像同步工作台”的职责。

### 3.3 失败显式暴露

当前设计理念不是“尽量帮用户悄悄兜底”，而是：

- 路径非法就报错
- 不是 Git 仓库就报错
- Git 命令失败就报错
- commit message 为空就报错

界面会显示错误横幅，不会伪装成成功，也不会偷偷跳过关键失败。

## 4. 设计理念

如果要让专业 AI 评估 UI/UX，下面这些是当前界面背后的核心设计出发点。

### 4.1 差异优先，而不是按钮优先

页面主标题下方已经明确写出当前原则：

> 主视角永远是差异文件列表，其次才是同步动作与目标仓库状态。

这意味着本产品把“即将发生什么变化”放在“用户可以点击什么按钮”之前。  
用户先看差异，再决定是否同步、提交、推送。

### 4.2 方向感必须持续可见

仓库镜像最容易出错的地方，不是不会点按钮，而是用户一时忘记当前方向。

所以当前页面在多个位置反复强化方向语义：

- 顶部 `Direction Chip` 持续显示 `源 → 目标`
- 工具栏里有 `A → B` / `B → A` 两个显式切换按钮
- 页面文案中直接使用“目标仓库状态”而不是模糊写“仓库状态”

### 4.3 风险不隐藏，但也不制造恐慌

右侧状态面板会告诉用户目标仓库是否干净、未提交数量、未跟踪数量。  
当目标仓库不干净时，页面不会禁止一切操作，但会给出明确提示：

- 同步前请先判断现有改动是否需要保留

这是一种“显式提醒 + 仍由用户决策”的模式，而不是替用户做所有判断。

### 4.4 页面不是 IDE，而是操作台

当前界面没有采用“左侧树 + 中间编辑器 + 底部日志”的 IDE 模式，而是更接近“控制台 / 仪表台”：

- 顶部是任务上下文
- 中部上方是控制区
- 中部下方是两个并列工作面板
- 底部是硬规则提示

这样做的目的，是让用户在一个屏幕中完成“确认上下文 -> 查看差异 -> 执行动作”。

### 4.5 视觉语言服务于风险识别

当前视觉没有走轻量办公白底风格，而是使用深色背景、玻璃卡片和高对比重点色，目的不是炫技，而是让以下信息更容易被扫描：

- 当前方向
- 差异类型
- 主操作按钮
- 错误 / 成功提示
- 目标仓库是否干净

## 5. 当前页面设计

下面是现在实际落地的页面结构，便于外部 AI 直接评估。

### 5.1 整体布局

页面由四个层级组成：

1. 顶部状态条
2. 仓库条
3. 双栏工作区
4. 底部状态条

桌面宽屏下采用双栏布局；窄屏下工作区会折叠为纵向排列。

### 5.2 顶部状态条

顶部区域现在更接近桌面工具的状态栏，而不是传统 Hero。包含：

- 产品标识 `RepoMirror`
- 当前方向标签
- 当前源仓库与目标仓库名称
- 目标仓库分支与状态摘要
- `刷新` / `保存` 图标按钮

这个区域承担三个职责：

- 告诉用户这是什么工具
- 告诉用户当前正在处理哪个方向
- 提供全局刷新与保存入口

### 5.3 仓库条

仓库条分成 A、B 两行，每行都直接展示：

- 仓库槽位标签
- 当前路径
- 当前分支
- 当前状态
- `选目录` 按钮

同时在第二行右侧保留两个全局动作：

- `交换仓库`
- `切换方向`

目标仓库会额外标出 `target`，让方向语义持续可见。

### 5.4 左侧差异面板

左侧是当前页面的核心视觉区域，权重比右侧更大。

包含：

- 标题与当前可见计数
- 搜索框，可按路径 / 类型 / 保护规则搜索
- 筛选按钮：
  - 全部
  - 新增
  - 修改
  - 删除
  - 受保护
- 表格化差异列表
  - `Type`
  - `Path`
  - `Rule`
  - `Size`

其中 `受保护` 项用于显式告诉用户“这里有差异，但目标端规则不允许同步”，避免它们被悄悄吞掉。

如果筛选或搜索后没有结果，会显示空状态提示。

### 5.5 右侧目标仓库状态面板

右侧面板聚焦“目标仓库是否适合继续执行后续动作”。

当前包含：

- 仓库名
- 分支名
- 干净 / 有未提交改动
- 未提交数量
- 未跟踪数量
- 当前同步汇总
  - 新增
  - 修改
  - 删除
  - 受保护
- 风险提示条
- 提交信息输入框
- `执行同步` / `提交` / `推送` 按钮

这里的交互语义非常明确：

- 左侧回答“会同步什么”
- 右侧回答“目标仓库现在是什么状态，以及接下来能做什么”

### 5.6 底部状态条

页面底部不是帮助文档区，而是一条持续存在的状态条，固定呈现：

- 三条硬规则提示
- 最近一次刷新时间
- 当前成功 / 失败消息

它承担的是“全局运行状态回显”，不是额外操作区。

## 6. 当前视觉风格

当前 UI 采用的是一种“深色终端式操作台”风格，关键词如下：

- 深色底板 + 低饱和描边
- 青绿色、金色、红色作为状态色
- 紧凑的桌面应用栏式布局
- `Inter` 作为界面主字体
- `JetBrains Mono` 用于路径、分支、计数和状态信息

具体视觉实现特点：

- 主背景几乎不滚动，强调“固定工作台”而不是网页
- 顶部、仓库条、面板、底部状态条都有明确分层
- 差异列表使用表格化布局而不是卡片堆叠
- 关键状态由颜色和字重共同区分，而不是只靠文案
- 视觉重点放在“扫描差异”和“目标状态”，而不是营销式装饰

这套风格希望传达的是：

- 不是开发者 IDE
- 也不是普通表单后台
- 而是一个需要用户做谨慎判断的桌面同步控制台

---

## 7. 适合被评估的 UI/UX 重点

如果你准备把这个项目交给专业 AI 评估，建议重点看这些问题：

- 当前“差异优先”的信息层级是否足够清晰
- 同步方向是否已经足够不容易误判
- 控制区按钮是否过密，是否需要更强主次关系
- 左右双栏是否符合用户的任务流
- 目标仓库有未提交改动时，提示强度是否合适
- 当前差异列表是否信息不足，例如是否需要显示更多上下文
- 当前深色风格是否提升了专注感，还是增加了阅读负担
- 移动到窄屏纵向布局后，信息优先级是否仍然成立

## 8. 开发与构建

```bash

```

```bash
cd frontend
npm run build
```

```bash
go test ./...
```

桌面构建：

```bash
wails build
```
