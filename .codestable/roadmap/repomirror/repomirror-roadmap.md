# RepoMirror 实现规格

## 1. 项目目标

`RepoMirror` 是一个 `Go + Wails` 的 Windows 桌面工具。

它只做三件核心事情：

1. 记住两个 Git 仓库目录（A 和 B）
2. 展示两个仓库之间的差异文件
3. 按方向执行镜像同步，并支持对目标仓库做提交和推送

这个工具不是给最终用户看的 PRD，也不是讨论稿。
这份文档的用途是：让后续 AI 或开发者读完后，能直接知道要做什么、不要做什么、界面长什么样、核心逻辑怎么落地。

## 2. 固定技术方案

- 语言：`Go`
- 桌面框架：`Wails`
- 平台：`Windows`
- Git 交互方式：直接调用本机 `git.exe`
- 配置存储：本地配置文件

不要再改成 `Fyne`、`Node`、`PowerShell`、`Tauri` 或其他方案。

## 3. 功能范围

## 3.1 必须做

- 选择仓库 A 和仓库 B
- 记住 A/B 路径，重启后自动恢复
- 支持 `A → B` 和 `B → A`
- 展示两个仓库的差异文件
- 差异文件要标记 `新增` / `修改` / `删除`
- 执行同步
- 展示目标仓库当前是否有未提交改动
- 后续支持对目标仓库执行 `commit`
- 后续支持对目标仓库执行 `push`

## 3.2 不做

- 不做日志系统
- 不做日志导出
- 不做操作历史
- 不做多项目配置管理
- 不做自动监听同步
- 不做复杂 Git 工作流
- 不做 merge / rebase / stash / cherry-pick
- 不做冲突解决界面
- 不做远程分支管理

## 4. 产品核心思路

这个工具的主界面核心，不是“按钮”，也不是“日志”。

主界面核心是：

- A 仓库和 B 仓库当前有哪些差异文件
- 每个差异文件属于 `新增` / `修改` / `删除`
- 目标仓库当前是否干净
- 用户现在是否适合执行同步
- 同步后是否需要立即提交和推送

也就是说，用户打开工具后，首先应该看到“差异”和“目标仓库状态”，而不是一大堆辅助信息。

## 5. 核心使用场景

### 场景 1：日常同步

1. 打开工具
2. 自动加载 A/B 仓库路径
3. 自动计算当前方向下的差异文件
4. 用户查看差异列表
5. 点击同步
6. 如果目标仓库有改动，界面明确展示
7. 用户决定是否继续提交和推送

### 场景 2：检查目标仓库是否适合同步

1. 打开工具
2. 选择同步方向
3. 工具显示目标仓库 Git 状态
4. 如果目标仓库存在未提交改动，界面提示
5. 用户先处理目标仓库，再决定是否同步

### 场景 3：同步后直接提交推送

1. 执行同步
2. 同步完成后刷新目标仓库状态
3. 显示本次被改动的文件
4. 用户输入提交信息
5. 点击提交
6. 点击推送

## 6. 同步规则

## 6.1 总原则

当方向为 `A → B` 时：

- A 是源仓库
- B 是目标仓库
- 把 A 中允许参与同步的文件镜像到 B
- B 中源端不存在、且不受保护的文件允许被删除

`B → A` 同理。

## 6.2 源仓库参与同步的文件

源仓库参与同步的文件集合定义为：

- Git 已跟踪文件
- Git 未忽略的工作区文件

同时强制排除：

- `.git` 目录全部内容
- 任意层级的 `.gitignore`

建议命令：

- `git ls-files --cached --others --exclude-standard -z`

拿到结果后，再过滤掉所有 `.gitignore` 路径。

## 6.3 目标仓库保护规则

目标仓库以下内容必须保护：

- `.git` 目录全部内容
- 任意层级的 `.gitignore`
- 被目标仓库 ignore 规则命中的路径

这些内容不能被覆盖、不能被删除。

## 6.4 为什么同时看两边规则

必须同时看两边：

- 源仓库决定“哪些文件可以输出”
- 目标仓库决定“哪些文件必须保留”

不能只按源仓库规则同步，否则会误伤目标仓库本地约定。

## 7. 差异文件定义

差异文件列表是主界面最重要的数据。

以 `A → B` 为例：

### 7.1 新增

文件存在于 A，可参与同步，但在 B 中不存在。

标记为：`新增`

### 7.2 修改

文件在 A 和 B 都存在，但内容不同。

标记为：`修改`

建议比较方式：

- 先比文件大小
- 再比修改时间
- 最终以文件内容哈希或字节比较为准

不要只靠时间戳判断。

### 7.3 删除

文件不存在于 A，但存在于 B，且该文件不受目标端保护。

标记为：`删除`

### 7.4 不显示的文件

以下内容不进入差异列表：

- `.git` 目录内容
- `.gitignore` 文件
- 被目标仓库保护的 ignore 文件
- 两端完全一致的文件

## 8. 目标仓库 Git 状态

目标仓库状态必须在界面中明确展示。

至少展示以下几类信息：

- 是否存在未提交改动
- 是否存在未跟踪文件
- 当前分支名
- 是否存在可推送提交（可选首版）
- 是否存在可拉取差异（可选首版，不强求）

首版最少必须实现：

- 当前分支名
- 是否干净
- 未提交文件数量
- 未跟踪文件数量

建议命令：

- `git status --short`
- `git branch --show-current`

## 9. 主界面布局

界面要尽量减少视觉占用，把空间让给差异文件列表。

布局原则：

- 顶部放最少的配置与方向信息
- 中间大区域放差异文件列表
- 右侧或下方放目标仓库 Git 状态和动作按钮
- 不要放大块日志面板

## 11. 按钮行为定义

### 11.1 `选目录`

- 选择仓库路径
- 自动检测是否为 Git 仓库
- 保存后写入本地配置

### 11.2 `交换`

- 交换 A/B 路径
- 重新计算当前方向
- 重新刷新差异列表和目标状态

### 11.3 `A → B` / `B → A`

- 切换同步方向
- 重新定义源和目标
- 重新刷新差异列表
- 重新刷新目标仓库状态

### 11.4 `执行同步`

- 按当前方向执行镜像同步
- 成功后刷新差异列表
- 成功后刷新目标仓库状态

### 11.5 `提交`

- 仅对当前目标仓库操作
- 弹出提交信息输入框
- 执行 `git add` + `git commit`
- 成功后刷新目标仓库状态

说明：

- `commit` 是后续功能，但架构现在就要预留。
- 首版可以先只保留按钮占位，后续实现。

### 11.6 `推送`

- 仅对当前目标仓库操作
- 执行 `git push`
- 成功后刷新目标仓库状态

说明：

- `push` 也是目标仓库操作，不涉及源仓库。

## 12. 本地配置

配置文件建议保存到：

- `%APPDATA%/RepoMirror/config.json`

建议结构：

```json
{
  "projectA": "E:/h5-credit-monorepo",
  "projectB": "E:/h5-bl",
  "direction": "A_TO_B",
  "windowWidth": 1280,
  "windowHeight": 820
}
```

## 13. 后端模块划分

建议目录：

```text
cmd/repomirror/main.go
internal/app/
internal/config/
internal/gitops/
internal/diff/
internal/syncer/
internal/ui/
```

模块职责：

- `config`：读取和保存本地配置
- `gitops`：封装 `git status`、`git branch`、`git add`、`git commit`、`git push`
- `diff`：计算当前方向下的差异文件
- `syncer`：执行同步
- `app`：组织服务接口，给 Wails 前端调用
- `ui`：前端界面

## 14. Git 操作边界

允许的 Git 操作：

- `git rev-parse`
- `git ls-files`
- `git status`
- `git branch --show-current`
- `git add`
- `git commit`
- `git push`

不允许的 Git 操作：

- `git reset --hard`
- `git clean -fd`
- `git checkout`（切分支）
- `git merge`
- `git rebase`
- `git stash`

## 15. 实现顺序

按下面顺序开发：

1. 配置持久化
2. A/B 路径选择与方向切换
3. Git 仓库检测
4. 差异文件计算
5. 目标仓库状态展示
6. 执行同步
7. 同步后自动刷新
8. 目标仓库提交
9. 目标仓库推送

## 16. 完成标准

以下条件都满足，才算完成：

- 启动后能恢复上次 A/B 路径
- 能切换 `A → B` / `B → A`
- 主界面能展示差异文件列表
- 差异项能标记 `新增` / `修改` / `删除`
- 目标仓库未提交状态可见
- 同步执行正确
- `.gitignore` 不参与同步
- `.git` 不受影响
- 同步后界面自动刷新
- 后续可以无缝接入 `commit` / `push`
